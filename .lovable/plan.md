فهمت المشكلة: حالياً لوحة الطلبات لا تعرف أن عميل جديد عبّى الفورم إلا بعد Refresh، لأن التحديث الموجود داخل التطبيق محلي فقط لنفس الصفحة/التبويب ولا يصل من صفحة العميل إلى لوحة الوكيل/الأدمن. كمان رابط النواقص `/r/...` يفتح صفحة شبه فارغة عندما لا يستطيع قراءة الطلب أو الملاحظات، لذلك لا تظهر التفاصيل ولا الكومنت.

الخطة التي سأنفذها بعد الموافقة:

1. جعل الطلبات تظهر تلقائياً بدون Refresh
   - أضيف Polling ذكي داخل `useRequestsLive` يجلب الطلبات كل عدة ثواني أثناء وجود المستخدم على لوحة الوكيل/الأدمن.
   - سيبقى التحديث المحلي الحالي موجوداً للاستجابة السريعة بعد أي تعديل من نفس الصفحة.
   - سأجعل التحديث آمن وخفيف: يتوقف عند مغادرة الصفحة، ولا يفتح كل الطلبات بالغلط إذا كان الفلتر غير جاهز.
   - النتيجة: إذا العميل عبّى طلب جديد من جواله، يظهر عند الوكيل/الأدمن تلقائياً خلال ثوانٍ بدون Refresh.

2. تحسين الأخطاء بدل “صفر/فارغ”
   - أضيف حالة `error` للهوك حتى لو فشل تحميل الطلبات لا تظهر اللوحة وكأنها “لا توجد طلبات”.
   - في `/agent` و `/admin` أعرض رسالة واضحة مثل “تعذر تحميل الطلبات، سيتم إعادة المحاولة تلقائياً”.

3. إصلاح رابط النواقص حتى يعرض التفاصيل والكومنت
   - السبب المحتمل من الكود أن صفحة `/r/$requestId` تستخدم `getRequest` العادي، وهذا يعتمد على صلاحيات القراءة العادية. رابط العميل غالباً بدون Login، لذلك لا يستطيع قراءة تفاصيل الطلب أو `request_notes`، فتظهر صفحة فيها العنوان ورقم الطلب فقط مثل الصورة.
   - سأضيف API داخلي عام وآمن مخصص للرابط العام: `/api/public/reupload-request`.
   - هذا المسار سيرجع فقط المعلومات اللازمة للعميل: رقم الطلب، اسم العميل إن وجد، النواقص المفتوحة والكومنتات المطلوبة، بدون كشف بيانات حساسة أو كل ملفات الطلب.
   - سأحدث صفحة `/r/$requestId` لتستخدم هذا المسار في القراءة، ثم ترفع الملفات كما هي.

4. التأكد من أن “طلب النواقص” يجهز الرابط بشكل صحيح
   - عندما يضيف الوكيل/الأدمن نقص من صفحة تفاصيل الطلب، سأضمن أن حالة الطلب تتحول إلى `reupload` إذا كان نوع الملاحظة “نقص”.
   - زر نسخ رابط النواقص سيبقى ينسخ `/r/{requestId}`، لكن الصفحة الآن ستعرض الكومنتات المطلوبة فعلاً.

5. تحسين واجهة صفحة النواقص
   - استخدام نصوص `reuploadPage` الموجودة بالترجمة بدل عناوين تفاصيل الطلب العامة.
   - إذا لا توجد نواقص مفتوحة، تظهر رسالة واضحة “لا توجد نواقص مطلوبة حالياً” بدل صفحة مبهمة.
   - إذا الطلب غير موجود أو الرابط غلط، تظهر رسالة “الطلب غير موجود”.

ملاحظات تقنية:
- لن أستخدم `localStorage` لتحديد صلاحيات أو بيانات حساسة.
- التحديث الفوري سيتم بالـ polling لأن Directus Realtime يحتاج WebSocket مفعّل على السيرفر الخارجي، وهذا قد لا يكون مضموناً حالياً. لاحقاً إذا WebSocket مفعّل يمكن استبداله أو دعمه بجانب polling.
- مسار النواقص العام سيرجع بيانات محدودة جداً فقط حتى لا نفتح تفاصيل الطلب كاملة لأي شخص معه الرابط.
